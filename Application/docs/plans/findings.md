# 调研发现

## 共识点
- “像本机一样打开远程电脑应用”必须具备本地窗口化能力，仅有整桌面串流做不到真正无缝。
- 业界成熟方案分为三层：整桌面流式、分窗口流式、应用级协议/绘图命令转发。
- 体验最好的通常是 RemoteApp/ICA/HDX 一类方案，但它们都建立在强平台绑定或重度协议栈之上。
- 对通用远控产品来说，先做用户态可落地的替代方案，再逐步收敛到真正的无缝窗口，是更稳妥的路线。

## 主要路线对比

### 1. 整桌面流式增强
- 代表：RustDesk/AnyDesk/TeamViewer 现有主路线。
- 优点：跨平台、成熟、延迟低、复用现有架构最多。
- 缺点：本质仍是“看远程屏幕”，不是“打开远程应用”。
- 适合作为：基线能力和过渡方案。

### 2. 分窗口流式 + 本地代理窗口
- 代表：RemoteApp/RAIL、Citrix 无缝窗口、部分虚拟桌面产品。
- 优点：最接近目标体验，可把远程应用表现成独立本地窗口。
- 缺点：要解决窗口生命周期、焦点、剪贴板、快捷键、拖放、DPI、Z-order 等整条链路。
- 适合作为：RustDesk 的中期主目标。

### 3. 应用级协议转发 / 绘图命令转发
- 代表：Xpra、Waypipe，以及更激进的图形命令流方案。
- 优点：低带宽、可实现真正本地化渲染。
- 缺点：强依赖平台图形栈，通用性差，工程复杂度最高。
- 适合作为：Linux 定向方案或研究参考，不适合作为近期通用路线。

## 分歧点与取舍
- 共识最好体验来自 RemoteApp/ICA 类方案；分歧在于是否值得一开始就追求这一终局。
- 最终取舍：不直接追完整 RemoteApp，而是先做“虚拟显示 + 窗口化承载”的过渡方案。
- 放弃原因：真正的本地代理窗口要跨越太多 OS 壳层问题，首版成本和风险都过高。

## 综合最优解

### 推荐路线
1. Windows First，先做单应用窗口化远控。
2. 远端为应用分配虚拟显示或受控显示区域，本地为其开独立窗口承载视频流。
3. 通过协议补充应用实例、显示索引、窗口标题、尺寸、焦点和输入路由信息。
4. 第二阶段再从“窗口承载器”升级到“本地代理窗口”，支持多窗口和更自然的任务栏/切换行为。

### 不推荐路线
- 不推荐直接投入通用图形命令转发。
- 不推荐一开始就做 Windows/macOS/Linux 同步推进。
- 不推荐把“无界鼠标跨屏”与“真正 RemoteApp”混为同一任务，它们相关但不是同一层能力。

## 决策记录
- 采纳自 Agent-1：RemoteApp/RAIL、Citrix、Xpra/Waypipe 的分类框架。
- 采纳自 Agent-2：窗口管理、焦点、IME、快捷键和 DPI 才是“像本机”的真正难点。
- 采纳自 Agent-3：整屏串流仍应是底座，窗口级增强是现实演进方向。
- 采纳自 Agent-4：RustDesk 应走 Windows MVP -> 多窗口增强 -> 长期 RemoteApp 的分阶段实施路线。
- 放弃“纯协议级本地重渲染优先”：因为对 RustDesk 当前能力栈不匹配。

## 与当前仓库的直接相关能力
- 已有虚拟显示基础，可作为 Windows MVP 支点。
- 已有桌面 GPU 纹理渲染基础，可承载本地窗口化展示。
- 已有多显示捕获与桌面捕获链路，可复用为窗口化远控底座。
