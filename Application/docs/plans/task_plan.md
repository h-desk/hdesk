# 任务计划：远程无界桌面控制方案调研

## 目标
调研可实现“像本机一样打开远程电脑应用”的远程无界桌面/无缝窗口方案，产出可比较、可执行的技术路线建议。

- [x] Phase 0: Coordinator 分析
- [x] Phase 1: 并行研究
- [x] Phase 2: 方案综合
- [x] Phase 3: 执行规划
- [ ] Phase 4: 待用户确认后进入实现
- [x] Phase 5: 验收与结论

## 综合结论
- 真正“像本机应用”的方案，本质上不是整桌面远控，而是远程应用本地窗口化。
- 可选路线主要分三类：整桌面流式增强、分窗口流式/代理窗口、应用级协议转发。
- 对 RustDesk 来说，最现实的路径不是直接做完整 RemoteApp/RAIL 兼容，而是先做 Windows 优先的“虚拟显示 + 单应用/少量应用窗口化容器”，再逐步补窗口枚举、输入焦点和本地代理窗口。
- 纯图形命令转发或应用级协议重放，理论体验最好，但工程复杂度过高，不适合作为通用远控软件的近期路线。

## 方案摘要
推荐采用分阶段路线：
1. 近期先做替代型体验方案：远端应用运行在虚拟显示上，本地用独立窗口承载该路视频流，实现“像单独应用一样打开”。
2. 中期增加远端窗口枚举、生命周期同步、焦点/剪贴板/快捷键路由，向真正的 RemoteApp 体验靠近。
3. 长期只在 Windows 上尝试真正的分窗口本地代理窗口，Linux 优先考虑借鉴或对接 Xpra/Waypipe 思路，而不是统一重造所有平台的应用层协议。

## 任务拆解
- [ ] 子任务 1：定义产品目标和边界 — 预期产出：区分“替代型无界桌面”和“真正 RemoteApp”的 PRD/技术定义
- [ ] 子任务 2：设计 Windows MVP 架构 — 预期产出：虚拟显示池、应用启动器、单窗口承载器、输入映射协议
- [ ] 子任务 3：验证远端应用绑定到虚拟显示 — 预期产出：PoC，证明指定应用可稳定显示到独立虚拟屏
- [ ] 子任务 4：设计本地代理窗口协议 — 预期产出：窗口创建/销毁/标题/尺寸/焦点消息模型
- [ ] 子任务 5：补齐体验链路 — 预期产出：剪贴板、快捷键、DPI、音频路由、多窗口切换策略
- [ ] 子任务 6：评估跨平台扩展 — 预期产出：Windows/macOS/Linux 分别的可行性和不做项清单

## 关键决策
- 采纳“Windows First”路线：因为 Windows 已有虚拟显示和桌面捕获基础，风险最低。
- 采纳“先替代、后原生”的策略：先交付接近效果的窗口化远控，再决定是否投入真正 RemoteApp。
- 放弃“直接做通用图形命令转发”：复杂度和兼容性风险远高于收益。
- 放弃“一步到位跨平台无缝窗口”：平台窗口系统差异过大，会拖慢首个可用版本。

## 风险与应对
- 风险：窗口焦点、IME、快捷键和 Z-order 体验不自然。
	应对：MVP 明确限制范围，只支持单远程应用前台交互，不承诺完整本地壳集成。
- 风险：虚拟显示驱动和应用定向显示不稳定。
	应对：先做受控应用清单和固定分辨率 PoC，再扩大兼容范围。
- 风险：多窗口后带宽和编码实例数快速上升。
	应对：优先单窗口或主窗口，后续再加分窗口捕获与 ROI 编码。
- 风险：macOS/Linux 成本远高于 Windows。
	应对：将其列为后续独立路线，不阻塞 Windows MVP。
